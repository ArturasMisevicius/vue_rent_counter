# Инструкция по конфигурации услуг

> Обновлено: 2026-06-14. Инструкция описывает текущее поведение Tenanto, сверенное с `ServiceConfiguration`, `BillingService`, `TariffResolver` и Filament-формой конфигурации услуг.

## Назначение

Конфигурация услуги говорит системе, какую строку начисления надо создать в счете для конкретного объекта. Одна конфигурация относится к одному объекту и одной услуге.

Этот модуль подходит для повторяющихся начислений: электричество, вода, отопление, вывоз мусора, интернет, обслуживание, бассейн, баня и другие регулярные услуги. Подробное описание для арендатора пишите в поле `Описание в счете`.

## Перед созданием конфигурации

1. Проверьте или создайте `Utility Service`.
   - В `Name` пишется понятное название услуги.
   - В `Description` пишется длинное бизнес-описание услуги.
   - `Service Type Bridge` должен совпадать с реальным типом услуги. Для услуг по счетчику от этого зависит поиск совместимого счетчика.

2. Проверьте `Provider` и `Tariff` только если цена должна приходить из тарифа поставщика.
   - Ручные поля `Unit Rate` и `Base Fee` в конфигурации имеют приоритет над тарифом.
   - Если цена должна браться из тарифа, оставьте ручные поля ставки пустыми.

3. Проверьте назначение арендатора и счетчики.
   - Генерация счетов берет только активные назначения за расчетный период.
   - Услуги по потреблению требуют совместимый счетчик и показания.

## Поля формы

| Поле | Что вводить | Частая ошибка |
| --- | --- | --- |
| Property | Квартира, комната, помещение или объект, куда должна попасть строка услуги. | Создать одну конфигурацию и ожидать, что она применится ко всем объектам. |
| Utility Service | Сама услуга: электричество, интернет, мусор, обслуживание, бассейн, баня. | Пытаться вместить все детали в название услуги. |
| Описание в счете | Длинный текст, который попадет в строку сгенерированного счета. | Оставить пустым для сложной услуги и ожидать, что tenant поймет начисление по короткому названию. |
| Provider | Необязательный поставщик. Нужен, когда тариф принадлежит поставщику. | Выбрать поставщика, но тариф от другого поставщика. |
| Tariff | Необязательный источник цены. | Заполнить тариф и ручные ставки, а потом ожидать, что тариф будет главным. Главными будут ручные ставки. |
| Pricing Model | Как система понимает количество в строке счета. | Выбрать модель по потреблению для услуги без счетчика или показаний. |
| Unit Rate | Ручная цена за единицу, день, месяц или расчетное количество. | Ввести отрицательное значение или заполнить поле, когда цена должна идти из тарифа. |
| Base Fee | Фиксированная сумма, которая добавляется один раз к строке. | Использовать как комментарий или количество. |
| Distribution Method | Как делится общая услуга. | Ожидать, что метод распределения сработает, когда `Shared Service` выключен. |
| Shared Service | Включать только когда один общий расход надо разделить между активными назначениями. | Включить для обычной услуги на конкретный объект и случайно разделить сумму. |
| Effective From | Дата, с которой конфигурация может использоваться. | Поставить дату позже конца расчетного периода счета. |
| Effective Until | Последняя дата действия. Можно оставить пустым для бессрочной конфигурации. | Создать пересекающиеся активные конфигурации для одной услуги и объекта. |
| Area Type | Рабочая заметка для начислений по площади. | Ожидать, что поле меняет формулу. Сейчас расчет использует площадь из назначения. |
| Custom Formula | Заметка для будущих правил формул. | Ожидать, что текст будет выполнен как формула счета. Сейчас он не вычисляется. |
| Active | Должна ли генерация счетов учитывать эту конфигурацию. | Оставить старую конфигурацию активной после замены. |

## Как сейчас считаются модели

- `Fixed Monthly` / `Flat`: количество равно `1`; сумма считается как `quantity * unit_rate + base_fee`.
- `Fixed Daily`: количество равно числу дней расчетного периода; сумма считается как `days * unit_rate + base_fee`.
- `Consumption Based`, `Tiered Rates`, `Hybrid`: требуют совместимые показания счетчиков; сейчас используют измеренное количество, найденную ставку и базовый сбор.
- `Time Of Use`: использует зоны только когда есть зоны и потребление по зонам; иначе падает обратно на обычный расчет по ставке и базовому сбору.
- `Custom Formula`: значение хранится в записи, но свободный текст формулы пока не выполняется генерацией счета.

## Условия для начисления по потреблению

Чтобы конфигурация по потреблению попала в счет:

1. У услуги должен быть `Service Type Bridge`, например water, hot water, electricity, heating или gas.
2. У объекта должен быть совместимый счетчик.
3. Должно быть одно подтверждаемое показание до начала расчетного периода.
4. Должно быть одно подтверждаемое показание внутри расчетного периода и не позже даты окончания периода.

Если чего-то не хватает, bulk preview пропускает арендатора/объект с причиной `ineligible_meter_readings`.

## Сценарий счета с запросом показаний

Текущий рабочий сценарий для счета по показаниям:

1. Админ или менеджер открывает период через `Open Reading Cycle`, либо cron запускает команду `billing:open-reading-invoice-cycle` по расписанию.
2. Система создает или обновляет `BillingPeriod` для organization и выбранных дат, например `May 2026`.
3. В `BillingPeriod` сохраняются `reading_submission_deadline`, `invoice_generation_date` и `payment_due_date`.
4. Система находит активные назначения tenant/property с активными счетчиками за этот период.
5. Для каждого подходящего назначения система создает пустой черновик счета с `automation_level = reading_request`, привязывает его к `BillingPeriod` и отправляет tenant уведомление.
6. Tenant открывает уведомление, вводит показания и отправляет форму.
7. Счет получает статус проверки `readings_submitted`, а админы и менеджеры с правом редактировать счета получают уведомление.
8. Проверяющий открывает счет и нажимает `Prepare from Readings`.
9. Система пересчитывает строки черновика из подтверждаемых показаний, меняет `approval_status` на `ready_for_review` и оставляет счет черновиком.
10. Проверяющий может исправить строки вручную и затем нажать `Finalize`.
11. После финализации tenant получает уведомление в портале со ссылкой на историю счетов. Отправка счета по email/PDF остается отдельным действием `Send Invoice`.

Связка периода нужна для контроля: tenant вводит показания не “вообще”, а в открытый счет конкретного `BillingPeriod`. Property, tenant, meters, tariffs и services определяются через активное назначение и конфигурации услуг на даты этого периода; результаты расчета затем попадают в строки счета и snapshot.

Сейчас `invoice.due_date` у пустого `reading_request` используется как deadline ввода показаний. Отдельный `BillingPeriod.payment_due_date` уже хранится для следующего шага, где финальный счет сможет иметь собственный срок оплаты после проверки показаний.

Важно: для автоматической подготовки нужны два подтверждаемых показания: одно до начала периода и одно внутри периода. Если пары нет, счет можно заполнить вручную.

Tenant не может отправлять показания в свободном режиме без открытого черновика `reading_request`. Если tenant открывает раздел показаний без активного запроса, портал показывает ожидание нового запроса; если ссылка ведет на уже отправленный или закрытый запрос, повторная отправка блокируется.

## Примеры для других услуг

Для интернета, вывоза мусора, ремонтного фонда, бассейна, бани или обслуживания:

- Длинное объяснение для арендатора пишите в `Описание в счете`.
- Общее описание шаблона услуги можно дополнительно держать в `Utility Service Description`.
- Для обычной регулярной цены используйте `Pricing Model = Fixed Monthly` или `Flat`.
- Если сумма начисляется прямо на объект, заполните `Unit Rate`.
- `Shared Service` включайте только когда общий расход надо разделить.
- `Distribution Method = Equal` подходит для равного деления, `Area` для деления по площади, `By Consumption` только когда распределение должно идти по показаниям.
- Не используйте `Custom Formula` как основное поле комментария, пока правила выполнения формул не реализованы.

## Разовая услуга без конфигурации

Если услуга нужна только в одном счете, не создавайте ради нее постоянную конфигурацию услуги. На странице создания счета выберите арендатора и период, затем нажмите `Добавить ручную услугу`. Система откроет блок строк счета и добавит одну пустую ручную строку.

В поле `Описание` этой строки можно писать длинный tenant-facing текст: что именно ремонтировалось, за какой период интернет, что входит в вывоз мусора, бассейн или баню. Это описание сохраняется в строке счета и показывается в просмотре счета. Максимальная длина описания — 4000 символов.

Если строк нужно несколько, после открытия блока используйте кнопку `Добавить строку вручную` внутри списка строк счета.

## Что надо обсудить

- Нужно ли разделить `Описание в счете` на два поля: текст для арендатора и внутреннюю заметку для администратора?
- Нужно ли блокировать пересекающиеся активные конфигурации для одного объекта и одной услуги?
- Нужно ли скрыть или отключить `Custom Formula`, пока выполнение формул не реализовано?
- Нужно ли сделать шаблоны услуг для ремонта, бассейна, бани, мусора, интернета и похожих услуг?
- Нужно ли настраивать общие расходы на уровне здания, а не повторять конфигурацию на каждый объект?

## Безопасный сценарий

Если есть сомнение, делайте максимально простую конфигурацию:

1. Выберите объект.
2. Выберите услугу.
3. Поставьте `Flat` или `Fixed Monthly`.
4. Заполните `Unit Rate`.
5. Если tenant должен видеть подробности, заполните `Описание в счете`.
6. Оставьте tariff, shared service, area type и custom formula пустыми.
7. Заполните `Effective From`.
8. Перед финализацией создайте preview счета.
